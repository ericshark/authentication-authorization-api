
# Recap of previous work

1. HTTP request sent → hits FastAPI app → event loop handles it → passes 
   through any middleware first. Middleware is used for cross-cutting concerns 
   that apply to every request: logging, timing/tracing, and CORS (tells the 
   browser which frontend domains are allowed to make requests to the API — 
   only relevant for browser-based requests, not Postman or mobile).

2. Middleware wraps the app like layers of an onion — last defined runs 
   closest to the route on the way in, and first on the way out. After 
   middleware the request hits the route. Dependencies run before the route 
   function — simple ones return a value, others use yield where everything 
   before yield is setup and everything after yield is cleanup that runs after 
   the route finishes.

3. For requests sent with JWT in the Authorization header, the token is 
   extracted via oauth2_scheme = OAuth2PasswordBearer(tokenUrl="..."). 
   tokenUrl is only used by FastAPI docs UI to know where to send the login 
   form — it has no effect on actual token extraction logic.

4. get_current_user() is a custom dependency that verifies the JWT, extracts 
   the user id from the payload, fetches the User from the DB and returns it 
   to the route.

5. JWT is a string of three base64-encoded parts: header.payload.signature. 
   Anyone can read all three parts — there is no encryption. The security 
   comes from the signature, which is an HMAC hash of the header and payload 
   using your secret key. Nobody without the secret key can produce a valid 
   signature, so any tampering with the payload invalidates it. Should always 
   be sent over HTTPS so nobody can intercept and reuse it.

6. Base.metadata.create_all() looks at Base.metadata — which automatically 
   collects info about every model that inherits from Base — and creates any 
   tables that don't exist yet. It has no awareness of changes to existing 
   tables.

7. Switched to Alembic because it tracks every schema change as versioned 
   migration files — like git for your database. It compares your models 
   against the real DB, generates Python code describing the change, and keeps 
   a history of all migrations. It stores the current version in an 
   alembic_version table so on a new DB it runs all migrations from the 
   beginning, and on an existing DB it only runs the ones not yet applied.
