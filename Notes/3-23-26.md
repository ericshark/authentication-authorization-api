# Event Loop & Concurrency

The event loop is the main thread running inside one Gunicorn worker process. Multiple workers can run on multiple cores, each with their own independent event loop — essentially separate instances of your app.
When the event loop receives an async request (e.g. an external API call or async DB query), it suspends that coroutine and moves on to other requests while waiting. Non-blocking — no thread needed.
When the event loop hits a sync/blocking function, it offloads it to a thread pool worker. The thread blocks waiting for the result, but the event loop stays free to handle other requests. Threads have overhead (~1-8MB stack each) which is why we don't just use them for everything.
CPU-heavy work (image processing, ML, etc.) should be offloaded to a Celery worker — a completely separate Python process with its own GIL. Redis acts as the message broker (queue) between your app and the Celery worker. Your app drops a job in Redis and returns immediately; the worker picks it up independently.
The GIL (Global Interpreter Lock) is per Python process, not per server. It means only one thread inside a given process can execute Python code at a time. It does NOT affect separate processes — which is why Celery workers (separate processes) achieve true parallelism. Threads are still useful for I/O because a thread releases the GIL while waiting for I/O.

# SQLAlchemy

models.py — defines classes that map to your database tables. SQLAlchemy uses these to translate between Python objects and database rows.
schemas.py — defines Pydantic classes that control what data comes in from requests and goes out in responses. Keeps your database shape separate from your API shape.
The engine manages a connection pool — a set of reusable, persistent connections to the database. This avoids the overhead of opening and closing a new connection on every request.
A session borrows one connection from the pool and provides it to your route function via dependency injection (Depends). The dependency runs before your function starts, hands in the session, and cleans it up after.
yield in a dependency means: run everything up to yield before the function, hand the yielded value in, then resume after yield once the function finishes (used for setup and teardown — e.g. opening and closing a session).
flush() sends pending changes to the database within the current transaction but does not commit them — the changes aren't permanently saved yet and can still be rolled back.
.scalars() unwraps SQLAlchemy's Row wrapper and returns the actual model objects directly. Not about deduplication — for that you'd use .unique() (needed when joins cause duplicate rows).

# ORDER OF SELECT: 
stmt = (
    select(User)          # what table/model
    .join(...)            # combine with other tables
    .where(...)           # filter rows
    .group_by(...)        # group for aggregates
    .having(...)          # filter groups
    .order_by(...)        # sort results
    .offset(...)          # skip N rows
    .limit(...)           # take N rows
)
# multiple conditions — AND
select(User).where(User.age > 18, User.is_active == True)
# or explicitly
from sqlalchemy import and_, or_, not_
select(User).where(and_(User.age > 18, User.is_active == True))

# OR
select(User).where(or_(User.role == "admin", User.role == "mod"))

# NOT
select(User).where(not_(User.is_banned == True))

# IN a list
select(User).where(User.id.in_([1, 2, 3]))

# NOT IN
select(User).where(User.id.not_in([1, 2, 3]))

# LIKE (pattern matching)
select(User).where(User.name.like("John%"))    # starts with John
select(User).where(User.name.ilike("john%"))   # case insensitive

# NULL checks
select(User).where(User.deleted_at == None)    # IS NULL
select(User).where(User.deleted_at != None)    # IS NOT NULL

# ORDER BY
select(User).order_by(User.created_at.desc())  # newest first
select(User).order_by(User.name.asc())         # A to Z
select(User).order_by(User.age.desc(), User.name.asc())  # multiple

# JOIN
select(Post).join(User, Post.author_id == User.id)

# with relationship already defined on model, shorthand
select(Post).join(Post.author)

# GROUP BY + aggregate
from sqlalchemy import func
select(User.role, func.count(User.id)).group_by(User.role)

# HAVING (filter after group by)
select(User.role, func.count(User.id))\
    .group_by(User.role)\
    .having(func.count(User.id) > 5)

# DISTINCT
from sqlalchemy import distinct
select(distinct(User.role))
